### B-2. PrintAgent 方向性2: 完成PDFパス直接印刷モードの追加（新規要件・2026/06/29記録）

> **採用方針（ユーザー決定 2026/06/29）= 方向性2**。SmtpAgent と思想を統一し、共通印刷基盤として他モジュールに開けるようにする。

**現状（確認結果）**: PrintAgent は「完成PDFを受け取って印刷する」仕様**ではない**。
- 送信側(Web)は `t_order_reports.PrintPayload`（**構造化JSONデータ** = `PrintPayloadDto` の ReportType + Data + プリンタ名/部数）を渡す。
- PrintAgent 自身が QuestPDF で **PDFを生成**（`PdfGeneratorService.GeneratePdf`）。帳票レイアウトは PrintAgent 内 `Documents/`（OrderApprovalDocument / ReceivingSlipDocument / FactoryInvoiceDocument の3種のみハードコード）。
- その後 SumatraPDF でサイレント印刷（`SilentPrintService.PrintPdf`、OutputType=1/3 のみ）。
- → 帳票デザインは **PrintAgent内部の責任**で、対応は定義済み3帳票のみ。任意PDFは扱えない。

**やりたいこと（方向性2）**: PrintAgent に「**完成PDFのフルパスを受け取り、そのままサイレント印刷するモード**」を追加する。
- 送信側（各モジュール）が自前でPDFを生成 → パスを渡すだけで印刷できる（**帳票デザインを送信側に委ねる**）。
- SmtpAgent（pdf_path を受け取り添付送信）と同じ思想に統一でき、共通印刷基盤として他モジュールから利用しやすくなる。

**想定設計（要詳細設計）**:
- ジョブに「PDF生成モード（従来=payloadから生成）」と「直接印刷モード（pdf_path指定）」の区別を持たせる（例: payload の ReportType に `raw_pdf` 等を追加、または pdf_path 列で分岐）。
- PrintJobWorker: pdf_path 指定時は `PdfGeneratorService` を通さず、そのパスのPDFを直接 `SilentPrintService.PrintPdf` に渡す。実在チェック・印刷後の扱い（保管/削除）を定義。
- 既存の payload→生成方式は並行維持（3帳票は現状のまま）。
- 共通化（CommonModule への投入ヘルパー移設＝SmtpAgent と同様）は別途検討（案B方向。当面は資材経由のまま並行運用も可）。

**着手条件**: SMTP連携（下記 I）が一段落してから、または印刷の共通化が必要になった時点。本項は要件記録のみ（今回は未着手）。

### I-3. SmtpAgent FAXテスト送信対応（発注書テスト段階で必要・2026/07/03記録）

> **記録のみ（今回未着手）**。発注書（order_approval）のテスト段階で、印刷（PrintAgent 経由）だけでなく **FAX 送信（SmtpAgent 経由）も同時にテスト対象**になる。OutputType=2/3 のグループは FAX 経路（`DispatchEnqueueService` → `ISmtpQueueService.EnqueueAsync` → `t_smtp_queue` → SmtpAgent）を通るため、テスト送信の安全対策を含めて改修方針を先に固めておく。

**前提（ユーザー提示・db_common_dev `m_smtp_config`）**: SmtpAgent は **config_key で処理判定**する。現状の config_key 行（貼り付けが崩れていたため列対応は要確認）:

| config_key | host | port | from_address（推定） | 備考 |
|---|---|---|---|---|
| `Material` | 172.16.128.81 | 25 | ...@faxmail.com | 資材の通常送信用（想定） |
| `fax` | 172.16.128.81 | 25 | ...@faxmail.com | FAX 用（想定） |
| `mail` | 172.16.128.81 | 25 | — | メール用（想定） |
| `test-fax` | 172.16.128.81 | 25 | `0064871033@faxmail.com` | **テスト送信先固定**（想定） |

（※正確な列 config_key/host/port/from_address/その他 は次回ユーザーに確認）

**改修・確認したい論点（今後）**:
1. **投入側の config_key 選択**: MaterialModule `FaxDispatchOptions.ConfigKey` がどの config_key を指すか。通常運用（`Material`/`fax`）とテスト（`test-fax`）を切り替える仕組み（`TestSendEnabled`/`TestFaxNumber` との関係）を整理する。
2. **テスト送信の安全対策**: 実 FAX 番号へ誤送信しないよう、テスト時は `test-fax`（固定番号 `0064871033@faxmail.com`）へ強制上書きする経路を確認・整備。`DispatchEnqueueService.ResolveRecipientForSend`（TestSendEnabled 時にダミー番号へ上書き）と m_smtp_config の `test-fax` 行の役割分担を明確化。
3. **発注書テスト時の同時経路**: OutputType=3（印刷+FAX）で、印刷（t_print_queue）と FAX（t_smtp_queue）の双方に正しく投入されること、かつ PDF が二重生成されないこと（→ dispatch spec タスク 10.2「二重生成回避」と連動）。
4. **SmtpAgent 側の config_key ルーティング**: `Material`/`fax`/`mail`/`test-fax` の使い分けと、資材 FAX がどの config_key を使うべきかの確定。

**着手条件**: PrintMonitor（印刷経路）の実装・確認が一段落し、発注書の印刷/FAX 同時テストを行う段階で着手。まずは印刷経路（PrintAgent + m_printer + /Common/PrintMonitor）を優先する。
