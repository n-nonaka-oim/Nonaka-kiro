### B-2. PrintAgent 方向性2: 完成PDFパス直接印刷モードの追加（新規要件・2026/06/29記録）

> **採用方針（ユーザー決定 2026/06/29）= 方向性2**。SmtpAgent と思想を統一し、共通印刷基盤として他モジュールに開けるようにする。

**現状（確認結果）**: PrintAgent は「完成PDFを受け取って印刷する」仕様**ではない**。
- 送信側(Web)は `t_order_reports.PrintPayload`（**構造化JSONデータ** = `PrintPayloadDto` の ReportType + Data + プリンタ名/部数）を渡す。
- PrintAgent 自身が QuestPDF で **PDFを生成**（`PdfGeneratorService.GeneratePdf`）。帳票レイアウトは PrintAgent 内 `Documents/`（OrderApprovalDocument / ReceivingSlipDocument / FactoryInvoiceDocument の3種のみハードコード）。
- その後 SumatraPDF でサイレント印刷（`SilentPrintService.PrintPdf`、OutputType=1/3 のみ）。
- → 帳票デザインは **PrintAgent内部の責任**で、対応は定義済み3帳票のみ。任意PDFは扱えない。

**やりたいこと（方向性2）**: PrintAgent に「**完成PDFのフルパスを受け取り、そのままサイレント印刷するモード**」を追加する。
- 送信側（各モジュール）が自前でPDFを生成 → パスを渡すだけで印刷できる（**帳票デザインを送信側に委ねる**）。
- SmtpAgent（pdf_path を受け取り添付送信）と同じ思想に統一でき、共通印刷基盤として他モジュールから利用しやすくなる。

**想定設計（要詳細設計）**:
- ジョブに「PDF生成モード（従来=payloadから生成）」と「直接印刷モード（pdf_path指定）」の区別を持たせる（例: payload の ReportType に `raw_pdf` 等を追加、または pdf_path 列で分岐）。
- PrintJobWorker: pdf_path 指定時は `PdfGeneratorService` を通さず、そのパスのPDFを直接 `SilentPrintService.PrintPdf` に渡す。実在チェック・印刷後の扱い（保管/削除）を定義。
- 既存の payload→生成方式は並行維持（3帳票は現状のまま）。
- 共通化（CommonModule への投入ヘルパー移設＝SmtpAgent と同様）は別途検討（案B方向。当面は資材経由のまま並行運用も可）。

**着手条件**: SMTP連携（下記 I）が一段落してから、または印刷の共通化が必要になった時点。本項は要件記録のみ（今回は未着手）。
